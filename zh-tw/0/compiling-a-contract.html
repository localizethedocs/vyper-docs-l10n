編譯合約¶

命令行編譯器工具¶

Vyper 包含以下用於編譯合約的命令行腳本：

  • vyper：編譯 vyper 合約或封存檔案

  • vyper-json：提供編譯器的 JSON 介面

備註

--help 旗標提供每個腳本使用方式的詳細說明。

vyper¶

vyper 提供編譯器的命令行存取。它可以生成各種輸出，包括簡單的二進位檔、AST、介面和來源碼映射。

要編譯合約：

$ vyper yourFileName.vy

使用 -f 旗標指定要傳回的輸出格式。使用 vyper --help 查看完整的輸出選項列表。

$ vyper -f abi,abi_python,bb,bb_runtime,bytecode,bytecode_runtime,blueprint_bytecode,cfg,cfg_runtime,interface,external_interface,ast,annotated_ast,integrity,ir,ir_json,ir_runtime,asm,opcodes,opcodes_runtime,source_map,source_map_runtime,archive,solc_json,method_identifiers,userdoc,devdoc,metadata,combined_json,layout yourFileName.vy

備註

自 0.2.0 起，編譯器的 opcodes 和 opcodes_runtime 輸出由於缺少 0 填充而一直傳回不正確的 opcode（已透過 PR 3735 修補）。如果您依賴這些函數進行除錯，請使用最新的修補版本。

-p 旗標允許您設定搜尋要匯入的介面檔案時使用的根路徑。如果未指定，則預設為目前工作目錄。請參閱 Searching For Imports 以取得更多資訊。

$ vyper -p yourProject yourProject/yourFileName.vy

儲存佈局¶

顯示合約的預設儲存佈局：

$ vyper -f layout yourFileName.vy

這會輸出一個 JSON 物件，詳細列出由編譯器決定的所有狀態變數位置。

覆寫合約的預設儲存佈局：

$ vyper --storage-layout-file storageLayout.json yourFileName.vy

--storage-layout-file 旗標的輸入格式必須與 vyper -f layout 命令的 .storage_layout 欄位格式相符。

vyper-json¶

vyper-json 提供編譯器的 JSON 介面。它期望一個 JSON 格式化的輸入，並以 JSON 格式化的輸出 傳回編譯結果。

從透過 stdin 提供的 JSON 進行編譯：

$ vyper-json

從 JSON 檔案進行編譯：

$ vyper-json yourProject.json

預設情況下，輸出會傳送到 stdout。要重定向到檔案，請使用 -o 旗標：

$ vyper-json -o compiled.json

匯入介面¶

vyper-json 按以下順序搜尋匯入的介面：

  1. 在輸入 JSON 的 interfaces 欄位中定義的介面。

  2. 從輸入 JSON 的 sources 欄位中的合約所生成的衍生介面。

請參閱 Searching For Imports 以取得 Vyper 匯入系統的更多資訊。

在線編譯器¶

Try VyperLang!¶

Try VyperLang! 是由 Vyper 團隊託管的 JupyterHub 實例，作為使用 Vyper 開發和測試合約的沙盒環境。它需要 GitHub 進行登入，並支援透過瀏覽器進行部署。

Remix IDE¶

Remix IDE 是一個編譯器和 JavaScript VM，用於以 Vyper 以及 Solidity 開發和測試合約。

備註

雖然 Remix IDE 的 Vyper 版本編譯器會定期更新，但可能比儲存庫 master 分支中的最新版本稍落後。請確保位元組碼與您本機編譯器的輸出相符。

編譯器最佳化模式¶

Vyper CLI 工具接受 "none"、"codesize" 或 "gas"``（預設）最佳化模式。可以使用 ``--optimize 旗標進行設定。例如，呼叫 vyper --optimize codesize MyContract.vy 將編譯合約並針對程式碼大小進行最佳化。粗略總結 gas 和 codesize 模式之間的差異，在 gas 最佳化模式下，編譯器會嘗試生成最小化 gas 使用的位元組碼（在某種程度上），包括：

  • 使用稀疏選擇器表，相對於程式碼大小更優化 gas

  • 內聯一些常數，以及

  • 嘗試展開一些迴圈，尤其是針對資料複製。

在 codesize 最佳化模式下，編譯器會努力透過以下方式最小化程式碼大小

  • 使用密集選擇器表

  • 將程式碼外聯（out-lining），以及

  • 對資料複製使用更多迴圈。

啟用實驗性程式碼生成¶

When compiling, you can use the CLI flag --experimental-codegen or its alias --venom to activate the new Venom IR. Venom IR is inspired by LLVM IR and enables new advanced analysis and optimizations.

設定目標 EVM 版本¶

編譯合約程式碼時，您可以指定要編譯的目標以太坊虛擬機版本，以存取或避免特定功能。您可以透過來源碼 pragma 或編譯器選項來指定版本。當您需要靈活性時（例如，方便在不同鏈上部署），建議使用編譯器選項；當您需要位元組碼可重現性時（例如，在區塊瀏覽器上驗證程式碼），則建議使用來源碼 pragma。

備註

如果編譯器選項指定的 EVM 版本與來源碼 pragma 衝突，將會引發例外且編譯不會繼續。

例如，將以下 pragma 新增到合約中表示應針對 EVM 的「prague」分叉進行編譯。

#pragma evm-version prague

警告

針對錯誤的 EVM 版本進行編譯可能導致錯誤、異常或失敗的行為。請確保使用相符的 EVM 版本，尤其是在執行私有鏈時。

透過 vyper CLI 編譯時，您可以使用 --evm-version 旗標指定 EVM 版本選項：

$ vyper --evm-version [VERSION]

使用 JSON 介面時，您可以在 "settings" 欄位中包含 "evmVersion" 鍵：

{
    "settings": {
        "evmVersion": "[VERSION]"
    }
}

目標選項¶

以下是支援的 EVM 版本列表，以及每個版本引入的編譯器變更。各版本之間不保證向後相容性。一般來說，編譯器團隊維持非正式政策，即編譯器將支援 3 年的硬分叉規則集，但此政策可能會根據情況重新檢視。

london¶
paris¶

  • block.difficulty 已被棄用，改為使用其新別名 block.prevrandao。

shanghai¶

  • PUSH0 opcode 由編譯器自動生成，取代 PUSH1 0

cancun¶

  • transient 關鍵字允許宣告存在於暫態儲存中的變數

  • 標記為 @nonreentrant 的函數使用 TLOAD/TSTORE 而非 SLOAD/SSTORE 進行保護

  • 對於大多數記憶體操作，編譯器將自動生成 MCOPY opcode。

prague(default)¶

控制警告¶

Vyper 允許透過 CLI 旗標 -Wnone 來抑制警告，或透過 -Werror 旗標將（所有）警告升級為錯誤。

$ vyper -Wnone foo.vy   # 抑制警告

$ vyper -Werror foo.vy   # 將警告升級成錯誤

完整性雜湊¶

為了幫助工具檢測兩個建置是否相同，Vyper 提供了 -f integrity 輸出，它會輸出合約的完整性雜湊。完整性雜湊遞迴定義為來源碼的 sha256 加上其依賴項（匯入）的完整性雜湊和儲存配置覆蓋（如果有提供的話）。

Vyper 封存檔¶

Vyper 封存檔是一個可編譯的輸入來源碼和設定組合。技術上來說，它是一個 ZIP 檔案，具有特殊結構使其可作為編譯器的輸入。它可以使用任何副檔名，但慣例是使用 .zip 或 .vyz 副檔名。它必須包含一個 MANIFEST/ 資料夾，具有以下目錄結構。

MANIFEST
├── cli_settings.txt
├── compilation_targets
├── compiler_version
├── integrity
├── settings.json
├── searchpaths
└── storage_layout.json [OPTIONAL]

  • cli_settings.txt 是生成此封存檔的編譯執行中所使用設定的文字表示。

  • compilation_targets 是以換行分隔的編譯目標列表。目前僅支援一個編譯

  • compiler_version 是用於生成此封存檔的編譯器版本的文字表示

  • integrity 是輸入合約的 完整性雜湊

  • searchpaths 是以換行分隔的此次編譯執行中使用的搜尋路徑列表

  • settings.json 是此次編譯執行中使用的設定的 JSON 表示。它與 cli_settings.txt 是 1:1 對應的，但兩者都提供，因為它們對不同的工作流程（通常分別用於手動和自動化）都很方便。

  • storage_layout.json 是要在此次編譯執行中使用的儲存佈局覆寫的 JSON 表示。此為選用項目。

Vyper 封存檔案可以透過請求 -f archive 輸出格式來產生。編譯器也可以使用 --base64 旗標以 base64 編碼形式產生封存檔。Vyper 編譯器可以直接接受 .vyz 和 base64 編碼的 Vyper 封存檔作為輸入。

$ vyper -f archive my_contract.vy -o my_contract.vyz  # 將封存檔寫入 my_contract.vyz
$ vyper -f archive my_contract.vy --base64 > my_contract.vyz.b64  # 將封存檔寫入為 base64 編碼文字
$ vyper my_contract.vyz  # 編譯 my_contract.vyz
$ vyper my_contract.vyz.b64  # 編譯 my_contract.vyz.b64

編譯器輸入與輸出 JSON 說明¶

提供 JSON 輸入/輸出是為了與 Solidity 相容，然而建議的方式是使用前述的 Vyper 封存檔。所謂的「標準 json」輸入可以使用 vyper -f solc_json 輸出格式從合約生成。

在可能的情況下，Vyper JSON 編譯器格式遵循 Solidity 的格式。

輸入 JSON 說明¶

以下範例描述了 vyper-json 的預期輸入格式。（註解通常不允許在 JSON 中使用，此處僅用於說明目的）。

{
    // Required: Source code language. Must be set to "Vyper".
    "language": "Vyper",
    // Required
    // Source codes given here will be compiled.
    "sources": {
        "contracts/foo.vy": {
            // Optional: keccak256 hash of the source file
            "keccak256": "0x234...",
            // Required: literal contents of the source file
            "content": "@external\ndef foo() -> bool:\n    return True"
        }
    },
    // Optional
    // Sources given here are made available for import by the contracts
    // that are compiled. If the suffix is ".vy", the compiler will expect
    // Vyper syntax. If the suffix is "abi" the compiler will expect an
    // ABI object.
    "interfaces": {
        "contracts/bar.vy": {
            "content": ""
        },
        "contracts/baz.json": {
            "abi": []
        }
    },
    // Optional
    // Storage layout overrides for the contracts that are compiled
    "storage_layout_overrides": {
        "contracts/foo.vy": {
            "a": {"type": "uint256", "slot": 1, "n_slots": 1},
            "b": {"type": "uint256", "slot": 0, "n_slots": 1},
        }
    },
    // Optional
    "settings": {
        "evmVersion": "prague",  // EVM version to compile for. Can be london, paris, shanghai, cancun or prague (default).
        // optional, optimization mode
        // defaults to "gas". can be one of "gas", "codesize", "none",
        // false  and true (the last two are for backwards compatibility).
        "optimize": "gas",
        // optional, whether or not the bytecode should include Vyper's signature
        // defaults to true
        "bytecodeMetadata": true,
        // optional, whether to use the experimental venom pipeline
        // defaults to false
        "experimentalCodegen": false,
        // the search paths to use for resolving imports
        "search_paths": [],
        // The following is used to select desired outputs based on file names.
        // File names are given as keys, a star as a file name matches all files.
        // Outputs can also follow the Solidity format where second level keys
        // denoting contract names - all 2nd level outputs are applied to the file.
        //
        // To select all possible compiler outputs: "outputSelection: { '*': ["*"] }"
        // Note that this might slow down the compilation process needlessly.
        //
        // The available output types are as follows:
        //
        //    abi - The contract ABI
        //    ast - Abstract syntax tree
        //    interface - Derived interface of the contract, in proper Vyper syntax
        //    ir - intermediate representation of the code
        //    userdoc - Natspec user documentation
        //    devdoc - Natspec developer documentation
        //    evm.bytecode.object - Bytecode object
        //    evm.bytecode.opcodes - Opcodes list
        //    evm.bytecode.sourceMap - Source mapping (useful for debugging)
        //    evm.deployedBytecode.object - Deployed bytecode object
        //    evm.deployedBytecode.opcodes - Deployed opcodes list
        //    evm.deployedBytecode.sourceMap - Deployed source mapping (useful for debugging)
        //    evm.methodIdentifiers - The list of function hashes
        //
        // Using `evm`, `evm.bytecode`, etc. will select every target part of that output.
        // Additionally, `*` can be used as a wildcard to request everything.
        // Note that the sourceMapFull.pc_ast_map is the recommended source map to use;
        // the other types are included for legacy and compatibility reasons.
        //
        "outputSelection": {
            "*": ["evm.bytecode", "abi"],  // Enable the abi and bytecode outputs for every single contract
            "contracts/foo.vy": ["ast"]  // Enable the ast output for contracts/foo.vy
        }
    }
}

輸出 JSON 說明¶

以下範例描述了 vyper-json 的輸出格式。註解當然不允許使用，此處僅用於說明目的。

{
    // The compiler version used to generate the JSON
    "compiler": "vyper-0.1.0b12",
    // Optional: not present if no errors/warnings were encountered
    "errors": [
        {
        // Optional: Location within the source file.
        "sourceLocation": {
            "file": "source_file.vy",
            "lineno": 5,
            "col_offset": 11
        },
        // Mandatory: Exception type, such as "JSONError", "StructureException", etc.
        "type": "TypeMismatch",
        // Mandatory: Component where the error originated, such as "json", "compiler", "vyper", etc.
        "component": "compiler",
        // Mandatory ("error" or "warning")
        "severity": "error",
        // Mandatory
        "message": "Unsupported type conversion: int128 to bool"
        // Optional: the message formatted with source location
        "formattedMessage": "line 5:11 Unsupported type conversion: int128 to bool"
        }
    ],
    // Optional: not present if there are no storage layout overrides
    "storage_layout_overrides": {
        "contracts/foo.vy": {
            "a": {"type": "uint256", "slot": 1, "n_slots": 1},
            "b": {"type": "uint256", "slot": 0, "n_slots": 1},
        }
    },
    // This contains the file-level outputs. Can be limited/filtered by the outputSelection settings.
    "sources": {
        "source_file.vy": {
            // Identifier of the source (used in source maps)
            "id": 0,
            // The AST object
            "ast": {},
        }
    },
    // This contains the contract-level outputs. Can be limited/filtered by the outputSelection settings.
    "contracts": {
        "source_file.vy": {
            // The contract name will always be the file name without a suffix
            "source_file": {
                // The Ethereum Contract ABI.
                // See https://github.com/ethereum/wiki/wiki/Ethereum-Contract-ABI
                "abi": [],
                // Natspec developer documentation
                "devdoc": {},
                // Intermediate representation (string)
                "ir": "",
                // Natspec developer documentation
                "userdoc": {},
                // EVM-related outputs
                "evm": {
                    "bytecode": {
                        // The bytecode as a hex string.
                        "object": "00fe",
                        // Opcodes list (string)
                        "opcodes": "",
                        // The deployed source mapping.
                        "sourceMap": {
                            "breakpoints": [],
                            "error_map": {},
                            "pc_ast_map": {},
                            "pc_ast_map_item_keys": [],
                            "pc_breakpoints": [],
                            "pc_jump_map": {},
                            "pc_pos_map": {},
                            // The deployed source mapping as a string.
                            "pc_pos_map_compressed": ""
                        }
                    },
                    "deployedBytecode": {
                        // The deployed bytecode as a hex string.
                        "object": "00fe",
                        // Deployed opcodes list (string)
                        "opcodes": "",
                        // The deployed source mapping.
                        "sourceMap": {
                            "breakpoints": [],
                            "error_map": {},
                            "pc_ast_map": {},
                            "pc_ast_map_item_keys": [],
                            "pc_breakpoints": [],
                            "pc_jump_map": {},
                            "pc_pos_map": {},
                            // The deployed source mapping as a string.
                            "pc_pos_map_compressed": ""
                        }
                    },
                    // The list of function hashes
                    "methodIdentifiers": {
                        "delegate(address)": "5c19a95c"
                    }
                }
            }
        }
    }
}

錯誤¶

每個錯誤都包含一個 component 欄位，指示錯誤發生的階段：

  • json：解析輸入 JSON 時發生的錯誤。通常是無效的 JSON 或缺少必要值的結果。

  • parser：解析合約時發生的錯誤。通常是無效的 Vyper 語法的結果。

  • compiler：編譯合約時發生的錯誤。

  • vyper：Vyper 內部發生的意外錯誤。如果您收到此類型的錯誤，請開啟一個 Issue。

您也可以使用 --traceback 旗標，在遇到錯誤時接收標準的 Python traceback。
