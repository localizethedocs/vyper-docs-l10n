抽象模組¶

抽象模組是一種特殊的 模組，它提供了可自訂其邏輯的插入點。這些插入點以 **抽象方法**（abstract methods）的形式存在，即使用 @abstract 修飾的 內部 方法。這些方法可以被覆寫（override）來提供自訂邏輯。以下是一個範例：

# base_token.vy
# This is the abstract module

balances: public(HashMap[address, uint256])

# Abstract method is declared here
@abstract
def _before_transfer(sender: address, recipient: address, amount: uint256):
    ...

@external
def transfer(recipient: address, amount: uint256):
    # and used here
    self._before_transfer(msg.sender, recipient, amount)
    self.balances[msg.sender] -= amount
    self.balances[recipient] += amount

base_token 模組定義了一個傳輸掛鉤 _before_transfer() 作為抽象方法。它在每次傳輸時被呼叫，但沒有實作——實作留給初始化此模組的人。這讓此模組專注於自訂邏輯執行於「何處」，而覆寫模組則決定它「做什麼」。

要提供實作，模組需要匯入並 initializes 抽象模組，然後為每個抽象方法提供 @override：

# pausable_token.vy

import base_token

initializes: base_token

exports: base_token.transfer

paused: public(bool)

@override(base_token)
def _before_transfer(sender: address, recipient: address, amount: uint256):
    assert not self.paused, "transfers are paused"

現在每次呼叫 base_token.transfer() 都會在轉移代幣之前檢查暫停旗標。覆寫在編譯時期解析——不涉及執行時期分派或繼承。

As you can see in the example above, what makes a module 「abstract」 is only the presence of abstract methods. Furthermore, that is the only difference, everything that can be done in a concrete (i.e. non-abstract) module can also be done in an abstract module.

抽象方法¶

抽象方法是使用 @abstract 修飾的 內部方法。其主體必須由省略符號字面值（...）組成，前面可以有文件字串（docstring）和註解。

@abstract
def an_abstract_method(x: uint256) -> uint256:
    """This is a docstring"""
    ...

抽象方法沒有其他限制，它們可以有任何簽章（signature），可以使用任何修飾器（只要是內部方法有效的修飾器即可），並且可以像具體方法一樣被呼叫：

@pure
@abstract
def name() -> String[10]:
    ...

@payable
@nonreentrant
@abstract
def foo(bar: DynArray[Bytes[20], 10]) -> DynArray[String[15], 38]:
    ...

def concrete_calls():
    _name: String[10] = self.name()
    complex_expression: String[15] = self.foo([]).pop()

覆寫抽象模組¶

由於抽象模組本質上是不完整的，因此需要另一個模組來完成它們，為其抽象方法提供實作：

import base_token

initializes: base_token

@override(base_token)
def _before_transfer(sender: address, recipient: address, amount: uint256):
    assert not self.paused, "transfers are paused"

若要讓模組 M1 覆寫模組 M2：

  1. M1 必須 初始化 M2

  2. M1 必須覆寫 M2 的每個抽象方法，請參閱 覆寫抽象方法

備註

因此不需要選擇要使用哪個覆寫，抽象模組只能被覆寫一次。這是由初始化系統保證的。

覆寫抽象方法¶

要覆寫模組 base_token 的抽象方法 _before_transfer，覆寫模組必須定義一個內部方法 _before_transfer，並加上 @override(my_abstract_module) 修飾器：

@override(base_token)
def _before_transfer(sender: address, recipient: address, amount: uint256):
    assert not self.paused, "transfers are paused"

由於對 base_token._before_transfer 的任何呼叫都會在編譯時期被替換為對覆寫的 _before_transfer 的呼叫，因此對前者的任何呼叫都必須是對後者的有效呼叫：

# abstract_m.vy
@pure
@abstract
def compute(x: uint256) -> uint256:
    ...

import abstract_m
initializes: abstract_m

# Valid: signature matches exactly
@pure
@override(abstract_m)
def compute(x: uint256) -> uint256:
    return x + 1

import abstract_m
initializes: abstract_m

# Invalid: parameter and return type don't match the ones in abstract_m.compute
@pure
@override(abstract_m)
def compute(x: uint8) -> uint8:
    return x + 1

import abstract_m
initializes: abstract_m

# Invalid: parameter name doesn't match
@pure
@override(abstract_m)
def compute(y: uint256) -> uint256:
    return y + 1

import abstract_m
initializes: abstract_m

# Invalid: Has looser mutability: nonpayable vs the original which is pure
@nonpayable
@override(abstract_m)
def compute(y: uint256) -> uint256:
    return y + 1

一個良好的經驗法則如下；覆寫必須具有：

  • 與抽象方法相同的參數：相同名稱、相同型別和相同預設值。

  • 與抽象方法相同的回傳型別。

  • 與抽象方法相同的修飾器，除了 @override 和 @abstract。

遵循上述規則將始終產生有效的覆寫。然而，這些規則在某些使用情境中可能過於嚴格，因此實際規則更加靈活：

  • 抽象方法的每個參數必須出現在覆寫中的相同位置，且使用相同的名稱。

  • 如果抽象方法為參數定義了預設引數，則預設引數的值在覆寫中必須相符。任何值都與省略符號（...）相符。

  • 覆寫可以在右側新增額外的參數，只要它們具有預設值即可。

  • 覆寫可以為抽象方法中的強制參數新增預設值。

  • 覆寫中的每個參數型別必須是抽象方法中對應參數型別的超型別（super-type）。

  • 覆寫的回傳型別必須是抽象方法回傳型別的子型別（sub-type）。

  • 覆寫的 可變性 必須與抽象方法的相同（或更嚴格）。

  • 覆寫的 可重入性 必須與抽象方法的完全相符。

呼叫抽象方法¶

抽象方法的呼叫方式與具體方法相同。此外，呼叫另一個模組的抽象方法需要使用 uses 陳述式。

# base_token.vy

@abstract
def _before_transfer(sender: address, recipient: address, amount: uint256):
    ...

@external
def transfer(recipient: address, amount: uint256):
    # call to abstract method in same module
    self._before_transfer(msg.sender, recipient, amount)
    self.balances[msg.sender] -= amount
    self.balances[recipient] += amount

import base_token

# required by the call below
uses: base_token

def simulate_transfer(sender: address, recipient: address, amount: uint256):
    # call to abstract method in different module
    base_token._before_transfer(sender, recipient, amount)

所有對抽象方法的呼叫都會在編譯時期解析為具體的覆寫：沒有執行時期分派。

當模組覆寫抽象方法時，編譯器要求呼叫者使用最具體的路徑。特別是，如果 self._before_transfer 覆寫了 base_token._before_transfer，則該模組內的任何呼叫都必須透過 self，而不是透過 base_token：

import base_token

initializes: base_token

@override(base_token)
def _before_transfer(sender: address, recipient: address, amount: uint256):
    assert not self.paused, "transfers are paused"

def foo():
    # Valid: uses the override directly
    self._before_transfer(msg.sender, msg.sender, 0)

    # Invalid: base_token._before_transfer is overridden by
    # self._before_transfer, call that instead.
    base_token._before_transfer(msg.sender, msg.sender, 0)

進階用法¶

在本節中，我們將解釋上述規範的一些可能不會立即想到的後果，這些在進階情境中很有用。

抽象的覆寫模組¶

覆寫模組本身也可以是抽象的。換句話說，@abstract 和 @override 可以共存於同一個模組中（甚至可以 在同一個方法上）：

# checked_token.vy

import base_token

initializes: base_token

@override(base_token)
def _before_transfer(sender: address, recipient: address, amount: uint256):
    assert self.check_address(sender), "Invalid sender"
    assert self.check_address(recipient), "Invalid recipient"
    assert self.check_cap(amount), "Invalid amount"

@abstract
def check_cap(amount: uint256) -> bool:
    ...

@abstract
def check_address(addr: address) -> bool:
    ...

這裡 checked_token.vy 為 base_token 提供了一個具體的 _before_transfer，但它本身是一個抽象模組，因為它引入了 check_cap 和 check_address。最終模組透過覆寫這些方法來完成鏈：

# my_token.vy
import checked_token

initializes: checked_token

MAX_AMOUNT: constant(uint256) = 10**24
total_supply: uint256

@override(checked_token)
def check_cap(amount: uint256) -> bool:
    return 0 < amount and amount <= MAX_AMOUNT

@override(checked_token)
def check_address(addr: address) -> bool:
    return addr != empty(address)

覆寫多個模組¶

單一模組可以初始化多個抽象模組，並為每個模組獨立提供覆寫。這就是組合不相關關注點的方式，例如將一個模組的傳輸驗證與另一個模組的手續費設定結合在一起。

使用來自 前面範例 的 base_token，考慮一個管理存取控制的第二個抽象模組：

# access_control.vy

@abstract
def _is_allowed(user: address) -> bool:
    ...

def check_allowed(user: address):
    assert self._is_allowed(user), "access denied"

合約可以初始化兩個模組並覆寫每個抽象方法：

# my_token.vy

import base_token
import access_control

initializes: base_token
initializes: access_control

exports: base_token.transfer

allowed: HashMap[address, bool]

@override(access_control)
def _is_allowed(user: address) -> bool:
    return self.allowed[user]

@override(base_token)
def _before_transfer(sender: address, recipient: address, amount: uint256):
    access_control.check_allowed(sender)

覆寫模組將兩個關注點交織在一起：_before_transfer 委派給存取控制模組來把關誰可以發送代幣。

覆寫多個抽象方法¶

當一個模組覆寫多個其他模組時，可能會發生抽象模組共用方法名稱的情況。為了處理這種情況，單一方法可以透過新增多個 @override 修飾器來覆寫多個方法。但請注意，該方法必須是每個抽象方法的 有效覆寫：

# minter.vy

import my_roles

@abstract
def get_role(user: address) -> my_roles.ROLE:
    ...

# authentication_provider.vy

import my_roles

@abstract
def get_role(user: address, default_role: my_roles.ROLE) -> my_roles.ROLE:
    ...

import minter
import authentication_provider
import my_roles
initializes: minter
initializes: authentication_provider

roles: HashMap[address, my_roles.ROLE]

# By having `default_role` as an optional parameter, it's a valid override of both
# `minter.get_role` which does not have that parameter, and
# `authentication_provider.get_role` which has it as a mandatory parameter

@override(minter)
@override(authentication_provider)
def get_role(user: address, default_role: my_roles.ROLE = empty(my_roles.ROLE)) -> my_roles.ROLE:
    role: my_roles.ROLE = self.roles[user]
    if role == empty(my_roles.ROLE):
        return default_role
    return role

延遲覆寫¶

在某些情況下，模組可能不想覆寫另一個模組的每個方法。如果是這種情況，可以組合 @abstract 和 @override 來延遲提供實作：

# base.vy

@abstract
def method_a() -> uint256:
    ...

@abstract
def method_b() -> uint256:
    ...

# middle.vy
import base

initializes: base

# Concrete override
@override(base)
def method_a() -> uint256:
    return 0

# Delegates
@abstract
@override(base)
def method_b() -> uint256:
    ...

# top.vy
import middle

initializes: middle

@override(middle)
def method_b() -> uint256:
    return 42

任何對 base.method_b 的呼叫都會解析為對具體實作的呼叫：top.method_b。

預設實作¶

有時需要為抽象方法提供預設實作，這可以透過定義一個包含此邏輯的單獨方法來完成：

# base_token.vy

@abstract
def _before_transfer(sender: address, recipient: address, amount: uint256):
    ...

def _before_transfer_default(sender: address, recipient: address, amount: uint256):
    assert sender != empty(address), "transfer from zero address"
    assert recipient != empty(address), "transfer to zero address"

這允許在下游接受預設實作，如下所示：

# simple_token.vy

import base_token

initializes: base_token

@override(base_token)
def _before_transfer(sender: address, recipient: address, amount: uint256):
    base_token._before_transfer_default(sender, recipient, amount)

甚至允許在使用預設實作的同時新增額外的邏輯：

# pausable_token.vy

import base_token

initializes: base_token

paused: public(bool)

@override(base_token)
def _before_transfer(sender: address, recipient: address, amount: uint256):

    # Use default check
    base_token._before_transfer_default(sender, recipient, amount)

    # Add custom logic
    assert not self.paused, "transfers are paused"

建議：

  1. 預設實作的名稱應與抽象方法相同，並在後面加上 _default。

  2. 預設實作不應該在任何地方被呼叫（除了在覆寫中），因為這會使將方法設為抽象的初衷失去意義。
